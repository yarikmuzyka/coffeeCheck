# coffeeCheck — Product Roadmap

Оновлено: 12 липня 2026

## Бачення продукту

coffeeCheck допомагає людям, які регулярно купують specialty coffee, памʼятати
улюблені лоти, покращувати рецепти заварювання та краще розуміти власні смакові
вподобання.

Основна продуктова гіпотеза:

> Користувачі регулярно вестимуть журнал кави й заварювань, а частина з них буде
> готова платити за необмежену історію, розширену аналітику та корисні персональні
> рекомендації.

## Принципи розвитку

- Спочатку безпека та ізоляція даних, потім зростання й монетизація.
- Кожен етап реалізується окремими невеликими змінами з тестами.
- Перехід до наступного етапу відбувається лише після виконання Definition of Done.
- Ліміти перевіряються на сервері; приховування кнопок у UI не є захистом.
- Дані понад ліміт ніколи не приховуються і не видаляються автоматично.
- Не ускладнювати billing і тарифи до підтвердження регулярного використання.

## Етап 1. Повноцінна авторизація

Ціль: замінити спільний Basic Auth на персональний вхід через Google OAuth.

### Реалізація

- [x] Підключити Auth.js із Google provider.
- [x] Додати Prisma-моделі `User`, `Account`, `Session` та `VerificationToken`.
- [x] Додати сторінку входу, кнопку виходу та короткий профіль користувача.
- [x] Додати `userId` до персональних сутностей: `Roaster`, `Coffee`, `BrewLog`.
- [x] Визначити `Process` як глобальний каталог або персональну сутність.
- [x] Додати helper, який повертає поточного авторизованого користувача.
- [x] Додати перевірку користувача в кожен read-запит і Server Action.
- [x] Підготувати двоетапну міграцію існуючих даних без втрати записів.
- [x] Привʼязати всі поточні дані до основного акаунта власника.
- [x] Зробити `userId` обовʼязковим після backfill.
- [x] Видалити Basic Auth лише після повної перевірки OAuth-потоку.
- [x] Оновити документацію та Vercel environment variables.

### Definition of Done

- Власник входить через Google та бачить усі наявні дані.
- Неавторизований відвідувач потрапляє на сторінку входу.
- Після виходу персональні сторінки та actions недоступні.
- Жоден запит до персональних даних не виконується без `userId`.
- Production build, unit-тести, integration-тести й CI успішні.

## Етап 2. Перевірка ізоляції двох акаунтів

Ціль: довести, що користувачі не можуть бачити або змінювати чужі дані.

### Ручний сценарій

- [ ] Увійти основним Google-акаунтом і створити тестові дані.
- [ ] Увійти другим тестовим акаунтом і перевірити порожній особистий простір.
- [ ] Створити окремі дані другим акаунтом.
- [ ] Спробувати відкрити прямі URL записів першого акаунта.
- [ ] Спробувати змінити та видалити чужі записи прямими запитами.
- [ ] Перевірити, що dashboard та analytics не змішують дані акаунтів.

### Автоматичні тести

- [ ] Користувач A не читає записи користувача B.
- [ ] Користувач A не змінює записи користувача B.
- [ ] Користувач A не видаляє записи користувача B.
- [ ] Вкладені сутності також перевіряють власника батьківського запису.
- [ ] Чужий ID повертає `404` або `403` без витоку назви чи інших даних.

### Definition of Done

- Усі ручні сценарії виконані для двох реальних акаунтів.
- Автоматичні security-тести проходять у CI.
- Немає Prisma-запитів до персональних сутностей без tenant-фільтра.

## Етап 3. Безкоштовний тариф і контроль використання

Ціль: дати достатньо цінності для перевірки продукту та обмежити витрати й
зловживання.

### Початкові ліміти Free

| Ресурс | Ліміт |
| --- | ---: |
| Обсмажчики | 5 |
| Кави | 20 |
| Заварювання | 50 |
| Аналітика | Базова |
| Експорт | Недоступний |

### Реалізація

- [ ] Додати поняття тарифу та entitlements користувача.
- [ ] Зберігати ліміти централізовано, а не дублювати magic numbers.
- [ ] Перевіряти ліміти в Server Actions перед створенням записів.
- [ ] Показувати використання: наприклад, `37 із 50 заварювань`.
- [ ] Показувати мʼяке попередження після 80% ліміту.
- [ ] На 100% блокувати лише створення нових записів.
- [ ] Залишати перегляд, редагування, експорт власних даних і видалення доступними.
- [ ] Видалення запису звільняє місце в ліміті.
- [ ] Додати зрозумілу сторінку порівняння Free та майбутнього Pro.
- [ ] Додати rate limiting для чутливих і mutation-запитів.
- [ ] Покрити boundary cases тестами: `limit - 1`, `limit`, `limit + 1`.

### Definition of Done

- Ліміти неможливо обійти прямим викликом Server Action.
- Користувач завжди бачить поточне використання й причину блокування.
- Дані не втрачаються після досягнення ліміту.
- Два акаунти мають незалежні counters.

## Етап 4. Закритий beta-тест

Ціль: перевірити, чи люди регулярно ведуть журнал до запуску оплати.

### Запуск

- [ ] Запросити перші 10–20 користувачів.
- [ ] Додати простий onboarding: обсмажчик → кава → перше заварювання.
- [ ] Додати канал для feedback і коротке опитування після першого тижня.
- [ ] Додати privacy policy, terms та можливість видалити акаунт і дані.
- [ ] Відстежувати лише мінімально необхідні продуктові події без вмісту нотаток.

### Основні метрики

- Реєстрація → створення першої кави.
- Реєстрація → створення першого заварювання.
- Частка користувачів із 3+ заварюваннями протягом перших 7 днів.
- Retention на 7-й та 30-й день.
- Частка користувачів, які досягли 50% і 100% Free-ліміту.
- Найчастіше використовувані функції та точки відмови.

### Головний сигнал

Користувачі повертаються без персонального нагадування та регулярно додають нові
заварювання. До появи цього сигналу billing не є пріоритетом.

### Definition of Done

- Є щонайменше 10 активованих beta-користувачів.
- Є дані за 2–4 тижні використання.
- Проведено щонайменше 5 коротких інтервʼю.
- Зрозуміло, за яку конкретну цінність користувачі потенційно готові платити.

## Етап 5. Мінімальна монетизація

Ціль: перевірити willingness to pay без складної тарифної системи.

### Початкова пропозиція

**Free**

- 5 обсмажчиків;
- 20 кав;
- 50 заварювань;
- базова аналітика.

**Supporter / Pro — орієнтир €2.99/місяць**

- необмежені записи в межах fair use;
- повна аналітика;
- експорт CSV/JSON;
- ранній доступ до нових функцій.

Ціна є гіпотезою і має перевірятися на beta-користувачах до фінального запуску.

### Реалізація

- [ ] Обрати платіжного провайдера з урахуванням податків і доступності в Україні.
- [ ] Створити один продукт і одну місячну ціну.
- [ ] Додати hosted checkout замість власної платіжної форми.
- [ ] Додати webhook із перевіркою підпису та idempotency.
- [ ] Зберігати `customerId`, статус і період підписки.
- [ ] Додати customer portal для скасування та зміни платіжних даних.
- [ ] Додати grace period після невдалої оплати.
- [ ] Після завершення Pro не видаляти дані: лише блокувати нові записи понад Free.
- [ ] Додати тести webhook, повторних подій і зміни тарифу.
- [ ] Перед прийманням платежів перевірити комерційні вимоги до Vercel-плану.

### Метрики монетизації

- Перегляд paywall → початок checkout.
- Початок checkout → успішна оплата.
- Free → Pro conversion.
- Скасування підписки та причина скасування.
- Місячна виручка проти витрат на інфраструктуру й платежі.

### Definition of Done

- Реальна тестова та production-оплата проходять end-to-end.
- Повторний webhook не дублює підписку або платіж.
- Доступ змінюється автоматично після оплати та скасування.
- Є перші платні користувачі або чіткий негативний результат гіпотези.

## Етап 6. Платна цінність після перевірки монетизації

Пріоритет визначається beta-даними, а не лише припущеннями.

Кандидати:

- [ ] Порівняння рецептів однієї кави.
- [ ] Графіки за методами, країнами, сортами та обробками.
- [ ] Персональні рекомендації на основі історії.
- [ ] Експорт і резервна копія.
- [ ] Історія зміни рецепта.
- [ ] Нагадування про давно відкриту каву.
- [ ] Фотографії пакетів.
- [ ] Розпізнавання етикетки після перевірки вартості й точності.

Фото та AI не починаємо, доки ручний журнал не демонструє стабільний retention.

## Послідовність виконання

```text
Google OAuth
  → міграція поточних даних
  → ізоляція двох акаунтів
  → security-тести
  → Free-ліміти та usage UI
  → beta на 10–20 користувачів
  → перевірка retention
  → один платний тариф
  → платні функції на основі даних
```
